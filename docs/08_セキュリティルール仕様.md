# HeyMate セキュリティルール仕様（Draft）

## 1. 目的
- Firestore Security Rulesで、利用者の権限外アクセスを防止する。
- 非マッチユーザー間の接触、個人情報漏えい、不正操作を防ぐ。

## 2. 前提
- 認証済みユーザーのみ基本アクセス可。
- 権限制御の根拠データは `matches` / `blocks` / `users` に保持する。
- ルールは「許可する条件のみ明示」のホワイトリスト方式。

## 3. コレクション別アクセス方針
### `users`
- Read: 公開プロフィール項目のみ許可
- Write: 本人のみ更新可
- 禁止: 本人確認ステータス等の運営管理項目を一般ユーザーが更新

### `verifications`
- Read: 本人 + 運営権限
- Write: 本人は申請作成のみ、審査結果更新は運営権限のみ

### `swipes`
- Create: 本人のみ
- Read: 本人のみ
- Update/Delete: 原則禁止

### `matches`
- Read: 当事者2名のみ
- Write: Cloud Functions経由のみ（クライアント直接更新禁止）

### `chatRooms` / `messages`
- Read/Write: マッチ当事者かつブロック関係がない場合のみ
- 禁止: 非マッチユーザー、第三者アクセス

### `reports`
- Create: 認証済みユーザー可
- Read: 作成者本人 + 運営権限
- Update: 運営権限のみ

### `blocks`
- Create/Delete: 本人のみ
- Read: 本人のみ

### `deletionRequests`
- Create: 本人のみ
- Read: 本人 + 運営権限
- Update: 運営権限のみ

## 4. 共通制約
- 書き込み時に `request.auth.uid == resource/userId` を検証。
- サーバー時刻（`request.time`）を利用し、クライアント時刻依存を避ける。
- 不要フィールドの書き込みを禁止（スキーマ制約）。

## 5. 監査・運用
- ルール違反アクセスは監査ログで検知できる構成にする。
- 重大違反パターンはCloud Functionsでアラート化する。

## 6. テスト観点
- 非マッチ状態でメッセージ送信不可
- 他人プロフィール更新不可
- 通報データの第三者閲覧不可
- ブロック後のチャット不可
