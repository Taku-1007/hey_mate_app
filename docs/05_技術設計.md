# HeyMate Technical Design (Draft)

## 1. 技術方針
- クライアント: Flutter（iOS / Android）
- 設計方針: Feature First + Clean Architectureの軽量適用
- 重要要件: 認証・安全性・監査ログをMVPから組み込む

## 2. 推奨アプリ構成（Flutter）
```text
lib/
  app/
    router/
    theme/
  features/
    auth/
    onboarding/
    profile/
    matching/
    chat/
    mypage/
    account/
    safety/
  shared/
    widgets/
    utils/
    constants/
```

## 3. 主要機能別コンポーネント
- `auth`: メール/Google/Appleログイン
- `onboarding`: 規約同意・年齢確認
- `profile`: 基本情報、ジムエリア、目標、本人確認ステータス
- `matching`: 候補取得、スワイプ、マッチ判定
- `chat`: マッチ後チャット、通知、ジム公開トグル
- `safety`: 通報、ブロック、凍結対象管理
- `account`: ログアウト、削除、問い合わせ導線

## 4. データモデル（初期）
- `User`
  - id, nickname, age, gender, homeCity, trainingGoals, strengths
- `Verification`
  - userId, idCheckStatus(pending/approved/rejected), facePhotoUrl, reviewedAt
- `MatchCandidate`
  - userId, candidateUserId, score, createdAt
- `Match`
  - id, userA, userB, matchedAt, status
- `ChatRoom`
  - id, matchId, gymShareConsentA, gymShareConsentB
- `Message`
  - id, roomId, senderId, text, sentAt
- `Report`
  - id, reporterId, targetUserId, reason, createdAt, status
- `Block`
  - id, userId, blockedUserId, createdAt
- `AccountDeletionRequest`
  - id, userId, requestedAt, status

## 5. API/バックエンド要件（抽象）
- 認証API: signup/login/token refresh
- プロフィールAPI: 取得/更新
- 本人確認API: 申請/審査ステータス取得
- マッチングAPI: 候補取得、Like/Skip、マッチ通知
- チャットAPI: ルーム作成、送受信、既読
- 安全API: 通報、ブロック、凍結連携
- アカウントAPI: 削除申請/実行、問い合わせ

## 6. セキュリティ/プライバシー設計
- 通信はTLS前提
- 認証トークンは安全なストレージで管理
- 個人情報は最小取得
- ログは目的限定で保存（調査/監査）
- アカウント削除時に個人データを削除フローへ移行

## 7. 通知設計
- マッチ成立通知
- 新着メッセージ通知
- 通知ON/OFF設定

## 8. テスト方針
- Unit: バリデーション、ドメインロジック
- Widget: 主要画面（認証、スワイプ、チャット）
- Integration: 登録→マッチ→チャットのE2E
- セキュリティ観点: 権限外アクセス、非マッチDM防止
