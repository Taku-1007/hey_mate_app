# HeyMate 技術設計（Clean Architecture + Firebase確定版）

## 1. 技術方針
- クライアント: Flutter（iOS / Android）
- 設計方針: Feature First + Clean Architecture
- バックエンド方針: Firebaseのみを利用（他BaaS/自前APIは採用しない）
- 重要要件: 認証・安全性・監査ログをMVPから組み込む

## 2. アーキテクチャ構成
各機能を以下の4レイヤーで分離する。

1. Presentation
- 画面、状態管理、入力バリデーション（UI観点）

2. Domain
- エンティティ、ユースケース、Repositoryインターフェース
- Firebase依存を持たない

3. Data
- Repository実装、DTO、Mapper
- Firebase SDKとの接続責務を持つ

4. Infrastructure
- Firebase初期化、Cloud Functions連携、Push通知などの外部連携

## 3. モジュール分割方針（責務ベース）
- 機能は「認証」「オンボーディング」「プロフィール」「マッチング」「チャット」「マイページ」「アカウント」「安全管理」に分割する。
- 各機能は `presentation` / `domain` / `data` の3層で実装する。
- `domain` はビジネスルールのみを扱い、Firebase SDKへの直接依存を禁止する。
- `data` は Firebase との入出力と、DTO-Entity変換を担当する。
- 共通UIや定数、汎用ユーティリティは機能外の共通モジュールに集約する。
- ルーティング、テーマ、DI、Firebase初期化、通知初期化はアプリ基盤モジュールとして分離する。
- 依存方向は `presentation -> domain -> data` を原則とし、逆依存はDIで解決する。

## 4. Firebase採用サービス
- Firebase Authentication
  - メール / Google / Appleログイン
- Cloud Firestore
  - ユーザー、マッチング、チャット、通報などの主要データ
- Cloud Functions
  - マッチ成立通知、通報処理、凍結処理、監査補助ロジック
- Firebase Cloud Messaging（FCM）
  - マッチ通知、新着メッセージ通知
- Firebase Storage
  - 顔写真、本人確認関連ファイル（必要最小限）
- Firebase App Check
  - 不正アクセス対策
- Firebase Analytics / Crashlytics
  - 利用分析、クラッシュ監視

## 5. 主要機能とFirebase責務
- `auth`: Authentication
- `onboarding`: Firestore（規約同意、年齢確認ステータス）
- `profile`: Firestore + Storage（プロフィール、顔写真）
- `matching`: Firestore + Functions（候補生成、Like/Skip、マッチ成立）
- `chat`: Firestore + FCM（チャット、通知、既読）
- `safety`: Firestore + Functions（通報、ブロック、凍結）
- `account`: Authentication + Firestore + Functions（退会、削除フロー）

## 6. データモデル（初期）
- `users`
  - uid, nickname, age, gender, homeCity, trainingGoals, strengths, visibility
- `verifications`
  - uid, idCheckStatus(pending/approved/rejected), facePhotoPath, reviewedAt
- `swipes`
  - fromUid, toUid, action(like/skip), createdAt
- `matches`
  - matchId, userA, userB, matchedAt, status
- `chatRooms`
  - roomId, matchId, gymShareConsentA, gymShareConsentB, updatedAt
- `messages`
  - messageId, roomId, senderUid, text, sentAt, readAt
- `reports`
  - reportId, reporterUid, targetUid, reason, createdAt, status
- `blocks`
  - blockerUid, blockedUid, createdAt
- `deletionRequests`
  - uid, requestedAt, status

## 7. セキュリティ設計
- Firestore Security Rulesで以下を強制
  - 非マッチユーザー間のチャットアクセス禁止
  - 本人以外のプロフィール更新禁止
  - 通報データへの一般ユーザーアクセス禁止
- App Checkを有効化
- 認証状態ベースでアクセス制御
- 個人情報は最小取得（住所詳細なし）

## 8. 運用・監査設計
- 通報、凍結、削除要求は監査用ログを保持
- Cloud Functionsで運営向けステータス更新を実装
- 利用規約同意と更新履歴を保存

## 9. テスト方針
- Unit: ユースケース、バリデーション、マッパー
- Widget: 主要画面（認証、スワイプ、チャット）
- Integration: 登録→マッチ→チャットのE2E
- ルール検証: Firestore Security Rulesのアクセス制御テスト
