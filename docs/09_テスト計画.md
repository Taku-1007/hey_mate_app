# HeyMate テスト計画（Draft）

## 1. 目的
- MVPの品質基準を定義し、リリース可否を判定できる状態にする。

## 2. テスト範囲
- 認証/オンボーディング
- プロフィール登録
- マッチング
- チャット
- 通報/ブロック
- アカウント削除
- Firestore Security Rules

## 3. テスト種別
### Unit Test
- バリデーション
- UseCaseロジック
- Mapper/変換

### Widget Test
- 主要画面の表示・入力・ボタンイベント
- エラー表示とリトライ導線

### Integration Test
- 登録→プロフィール→スワイプ→マッチ→チャット→通報 の一連フロー
- 退会・削除依頼フロー

### Security Rule Test
- 非マッチユーザーのメッセージ送信拒否
- 他ユーザー情報の不正更新拒否
- 通報データの第三者閲覧拒否

## 4. 受け入れ基準（抜粋）
- 未同意では利用開始できない
- 18歳未満は利用不可
- マッチ前チャット不可
- 通報機能が2タップ以内で到達可能
- アカウント削除依頼が正常登録される

## 5. テストデータ方針
- テストユーザーをロール別に用意（一般、違反者、運営）
- 本番データは使用しない
- PIIはダミーデータのみ

## 6. 実施タイミング
- PR時: Unit/Widget
- 日次: Integration（主要シナリオ）
- リリース前: 総合テスト + 手動受け入れ

## 7. レポート
- 不具合票に再現手順・期待値・実績値を記録
- 重大度（P0〜P3）を付与

## 8. リリース判定
- P0/P1未解決ゼロ
- 必須受け入れ基準100%達成
- セキュリティテスト重大NGゼロ

## 9. 未決定事項
- CIでの自動実行範囲
- E2Eの実行環境（実機/エミュレータ）
